# REPORT

## Текущий уровень зрелости API

Текущий API можно отнести к **1 уровню зрелости по модели Ричардсона**.

### Почему это 1 уровень:
- сервис уже разделён на несколько endpoint, а не использует один универсальный URL;
- используются отдельные адреса для разных действий: `/room`, `/add-room`, `/booking`;
- однако API ещё не является RESTful в полном смысле:
  - действия зашиты в названия endpoint;
  - HTTP-методы используются не как основное средство выражения операции над ресурсом;
  - названия ресурсов не унифицированы.

## Что можно улучшить

### 1. Названия endpoint
Текущие endpoint:
- `GET /room`
- `POST /add-room`
- `POST /booking`

Проблемы:
- `/add-room` описывает действие, а не ресурс;
- `/room` лучше называть во множественном числе, так как возвращается список комнат;
- структура API не единообразна.

### 2. Более RESTful вариант контракта

Предлагается заменить контракт на следующий:

- `GET /rooms` — получить список доступных комнат
- `POST /rooms` — добавить новую комнату
- `POST /bookings` — создать бронирование

### 3. Параметры запроса
Для поиска подходящих комнат можно сохранить query-параметры, но сделать endpoint более понятным:

Пример:
- `GET /rooms?checkIn=20210308&checkOut=20210311&guestsNum=2`

Это лучше, чем `GET /room`, потому что возвращается коллекция комнат.

### 4. Имена полей
В текущем контракте используются поля:
- `roomId`
- `guestNum`

С точки зрения JSON API это допустимо, но желательно придерживаться единого стиля именования во всём контракте.  
Для совместимости можно оставить camelCase, так как он уже используется в текущих тестах и коллекции.
